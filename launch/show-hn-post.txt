POST BODY:

Hi HN, I'm a solo developer (and veterinarian by day) who's been building a multi-agent AI orchestration system. The recurring problem I kept hitting: long-running agent sessions fill up the context window and degrade or crash. The existing solutions all use LLM summarization, which (1) costs extra model calls, (2) adds latency, (3) is non-deterministic, and (4) busts your provider prompt cache every time it rewrites the prefix.

So I built a deterministic approach. Context Warp Drive folds old turns into compact structural skeletons, conserves the exact identifiers agents need (UUIDs, file paths, SHAs, ports) in a Coordinate Closet, freezes the folded prefix so it's reused byte-identical while the provider cache is warm, and pages folded content back in automatically when the agent touches a path again.

No model calls. No truncation. Cache stays hot.

Production telemetry from a real agent marathon (measured from the provider's own usage ledger, not estimated):
- 92.6% cache-read hit rate across 954 tool calls
- 1h49m continuous run, zero reorientation tax at epoch boundaries
- $56 total cost for the full session
- 173 recall suppressions vs 27 injections (the engine filters aggressively)

Offline benchmark (reproducible, no API key):
- CWD: $0.0208, 0 extra calls, 94% retention
- Truncation: $0.0516, 0 calls, 44% retention
- LLM Summarization: $0.0685, 6 calls, 44% retention

It's provider-agnostic (Anthropic content blocks, OpenAI tool_calls, Gemini parts) and the core engine has zero runtime dependencies.

Repo + reproducible benchmark: https://github.com/dogtorjonah/context-warp-drive

I'd love feedback on the architecture — especially from anyone who's dealt with context management for long-running agents. Happy to answer questions about the fold mechanics, the Coordinate Closet identifier conservation, or the cache economics.
